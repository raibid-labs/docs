Common Patterns Observed

Project Structure

Most raibid-labs projects follow consistent patterns:

repo/
├── src/              # Source code
├── docs/             # Documentation
├── scripts/          # Automation scripts (often nushell)
├── tests/            # Test suites
├── .github/          # CI/CD workflows
├── justfile          # Build automation
├── README.md         # Project overview
└── config/           # Configuration files

Documentation Organization

Projects with significant documentation use:

  • Dedicated /docs directory
  • Markdown format
  • Categorized by topic
  • Often integrated with main documentation hub

Repository Themes

  1. Experimentation Focus: Many “hack-*” prefixed repos for exploration
  2. Infrastructure Automation: Heavy focus on CI/CD and DevOps
  3. TUI Applications: Multiple terminal UI projects (ratatui-based)
  4. DGX Integration: Specialized tooling for NVIDIA DGX systems
  5. Developer Experience: Tools to improve development workflows

Key Repository Deep Dives

raibid-ci

Purpose: Terminal-based management interface for ephemeral, auto-scaling CI/CD infrastructure on NVIDIA DGX Spark.

Tech Stack:

  • Rust with Ratatui for TUI
  • k3s for Kubernetes
  • Gitea for Git + container registry
  • Redis Streams for job queuing
  • KEDA for autoscaling
  • Flux for GitOps

Key Features:

  • Interactive TUI dashboard
  • Real-time monitoring
  • Job management (list, cancel, retry)
  • Agent lifecycle management
  • GitHub repository mirroring
  • YAML/TOML configuration

Build Tools:

  • Cargo for compilation
  • justfile for task automation
  • Tiltfile for local development
  • docker-compose for services

docs

Purpose: Centralized documentation aggregator combining content from multiple raibid-labs repositories.

Tech Stack:

  • Quartz v4 for static site generation
  • Nushell for automation
  • Git submodules for multi-repo aggregation
  • Obsidian-compatible vault structure

Key Features:

  • Automatic discovery of repos with /docs directories
  • Daily synchronization via GitHub Actions
  • Configurable ignorelist for filtering
  • Dual-purpose: local Obsidian vault and published website

Automation Scripts (Nushell):

  • discover-repos.nu - Find and catalog repositories
  • Repository filtering by patterns
  • Version tracking via releases/tags
  • Documentation detection

workspace

Purpose: Organization-wide workspace management scripts.

Automation Scripts:

  • analyze-org.sh - Organization analysis
  • init-repo.sh - Repository initialization
  • setup-github-actions.sh - CI/CD setup
  • sync-to-repos.sh - Cross-repo synchronization
  • validate-repo.sh - Repository validation

Organizational Conventions

Build Automation (justfile)

Raibid-labs projects use comprehensive justfiles with standardized recipes:

Build Commands:

  • build / build-release - Development and production builds
  • clean - Remove artifacts
  • Component-specific builds (cli, server, agent, etc.)

Test Commands:

  • test - Run all tests
  • test-verbose - Tests with output
  • test-unit / test-integration / test-doc - Specific test types
  • test-coverage - Coverage reports
  • Component-specific tests

Quality Commands:

  • check - Validate compilation
  • lint / lint-fix - Static analysis
  • fmt / fmt-check - Code formatting
  • ci - Run all quality checks

Development Commands:

  • watch - Auto-rebuild on changes
  • watch-test - Auto-rerun tests
  • Component-specific watchers

Utility Commands:

  • deps - Dependency tree
  • outdated - Check for updates
  • docs - Generate documentation
  • bench - Performance benchmarks

Docker/Registry Commands:

  • docker-build-* - Container builds
  • registry-* - Registry operations

Kubernetes Commands:

  • tk-* - Tanka operations (show, diff, apply)

Development Environment:

  • dev - Start local environment
  • quick-start - Initialize setup

Automation Scripts (Nushell)

Nushell is the preferred scripting language for automation:

Common Patterns:

  • GitHub API integration via gh CLI
  • JSON/TOML configuration files
  • Table-formatted output
  • Pattern-based filtering
  • Error handling with verbose flags

Script Categories:

  • Repository discovery and enumeration
  • Cross-repository operations
  • CI/CD automation
  • Validation and analysis

Documentation Standards

  • Markdown format for all documentation
  • /docs directory in project root
  • README.md with project overview
  • Separate guides for:
    • Architecture
    • Setup/Installation
    • Usage
    • Development
    • API/Reference

CI/CD Patterns

  • GitHub Actions for automation
  • Daily scheduled jobs for maintenance
  • Quality checks in pull requests
  • Container builds and registry pushes
  • Multi-architecture support (ARM64 + x86_64)

Implications for raibid-cli

Alignment with Conventions

raibid-cli should follow established patterns:

  1. Rust + Ratatui for TUI implementation
  2. Comprehensive justfile with all standard recipes
  3. Nushell scripts for GitHub API operations
  4. Structured /docs directory
  5. GitHub Actions for CI/CD
  6. Component-based architecture (lib + cli)

Integration Opportunities

  • Use GitHub API via gh CLI (like docs repo)
  • Integrate with raibid-ci for CI/CD operations
  • Support raibid-labs documentation structure
  • Provide organization-wide utilities
  • Enable cross-repository operations

Differentiation

raibid-cli serves as meta-management layer:

  • Organization-level view and operations
  • Repository lifecycle management
  • Consistent tool across all repos
  • Integration point for other raibid-labs tools
