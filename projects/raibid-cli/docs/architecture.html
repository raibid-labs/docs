raibid-cli Architecture

Overview

raibid-cli is a Rust-based terminal user interface (TUI) application for meta-management of repositories within the raibid-labs GitHub organization. It combines organization-wide repository operations with an interactive, keyboard-driven interface.

Design Philosophy

Core Principles

  1. Developer-First Experience

    • Keyboard-driven navigation
    • Real-time feedback
    • Minimal context switching
    • SSH-friendly (terminal-based)

  2. Organization-Level Perspective

    • Unified view of all repositories
    • Batch operations across multiple repos
    • Organization-wide insights

  3. Safe by Default

    • Detect uncommitted changes
    • Confirmation for destructive operations
    • Rollback capabilities
    • Detailed status reporting

  4. Extensible Foundation

    • Plugin-like command structure
    • Reusable core library
    • Configuration-driven behavior

Component Breakdown

1. TUI Layer (ratatui)

Responsibilities:

  • Render interactive terminal interface
  • Handle keyboard input
  • Manage application state
  • Display real-time updates
  • Navigate between views

Key Views:

  • Repository list with metadata
  • Sync progress view
  • Repository details
  • Settings/configuration
  • Help screen

Event Loop:

User Input → Event Handler → State Update → Render → Repeat

2. CLI Commands (clap)

Responsibilities:

  • Parse command-line arguments
  • Dispatch to appropriate handlers
  • Provide non-interactive operations
  • Enable scriptable workflows

Command Structure:

raibid-cli
├── sync              # Synchronize repositories
│   ├── --all        # Sync all repos
│   ├── --filter     # Sync matching repos
│   └── --concurrency # Parallel operations
├── clone             # Clone repositories
│   ├── --all        # Clone all repos
│   └── --filter     # Clone matching repos
├── list              # List repositories
│   ├── --format     # Output format (json, table, etc.)
│   └── --filter     # Filter criteria
├── tui               # Launch interactive TUI
└── config            # Configuration management
    ├── init         # Initialize config
    └── edit         # Edit config

3. Core Library (raibid-core)

Modules:

github.rs - GitHub API Integration

  • List organization repositories
  • Repository metadata retrieval
  • Rate limit management
  • Authentication handling

git.rs - Git Operations

  • Clone repositories
  • Pull updates
  • Detect repository state
  • Branch management
  • Remote operations

config.rs - Configuration Management

  • TOML/YAML configuration parsing
  • Environment variable overrides
  • Default settings
  • Validation

filter.rs - Repository Filtering

  • Pattern matching (glob, regex)
  • Metadata-based filtering
  • Exclusion lists
  • Custom filter expressions

sync.rs - Synchronization Engine

  • Concurrent operation management
  • Progress tracking
  • Error recovery
  • State persistence

types.rs - Shared Data Types

  • Repository metadata
  • Sync status
  • Configuration structures
  • Error types

Data Flow

Synchronization Flow

1. User initiates sync
   ↓
2. Load configuration
   ↓
3. Query GitHub API for repo list
   ↓
4. Apply filters to determine target repos
   ↓
5. For each repository (concurrent):
   a. Check if local clone exists
   b. If no: clone repository
   c. If yes: check local state
   d. Pull updates if safe
   e. Report status
   ↓
6. Aggregate results
   ↓
7. Display summary

Technology Stack

Core Dependencies

CLI Framework:

  • clap v4.x - Command-line argument parsing
    • Derive API for ergonomic definitions
    • Subcommands for command structure
    • Argument validation

TUI Framework:

  • ratatui v0.27.x - Terminal user interface
    • Immediate mode rendering
    • Flexible layouts
    • Rich widgets (lists, tables, paragraphs, etc.)
    • Event handling

Git Integration:

  • git2 v0.18.x - Libgit2 bindings
    • Repository operations
    • Clone, fetch, pull
    • Branch management

GitHub API:

  • gh CLI - GitHub API access
    • Organization queries
    • Repository enumeration
    • Uses existing authentication

Configuration:

  • serde v1.x - Serialization/deserialization
  • toml v0.8.x - TOML parsing
  • serde_yaml v0.9.x - YAML parsing

Async Runtime:

  • tokio v1.x - Asynchronous runtime
    • Concurrent operations
    • Async I/O
    • Task spawning

Error Handling:

  • anyhow v1.x - Flexible error handling
  • thiserror v1.x - Custom error types

Utilities:

  • crossterm v0.27.x - Terminal manipulation
  • dirs v5.x - Standard directory locations
  • chrono v0.4.x - Date/time handling

Configuration

Configuration File Structure

Location: ~/.config/raibid-cli/config.toml

Environment Variable Overrides

  • RAIBID_CLI_ORG - Override organization
  • RAIBID_CLI_WORKSPACE - Override workspace root
  • RAIBID_CLI_CONCURRENCY - Override concurrency
  • GITHUB_TOKEN - GitHub authentication token

Concurrency Model

Parallel Repository Operations

Main Thread
    ├─ TUI Event Loop (continuous)
    └─ Sync Coordinator
         ├─ Worker Pool (configurable size)
         │    ├─ Worker 1 → Repository A
         │    ├─ Worker 2 → Repository B
         │    ├─ Worker 3 → Repository C
         │    └─ Worker N → Repository N
         └─ Progress Aggregator
              └─ Update TUI State

Rate Limiting

  • GitHub API rate limits respected
  • Configurable local concurrency limits
  • Backoff strategy for failures
  • Progress reporting with ETA

Error Handling Strategy

Error Categories

  1. Fatal Errors - Stop execution

    • Invalid configuration
    • Missing authentication
    • GitHub API unavailable

  2. Recoverable Errors - Log and continue

    • Individual repository sync failures
    • Network timeouts (with retry)
    • Merge conflicts (skip with warning)

  3. Warnings - Log but don’t fail

    • Uncommitted changes detected
    • Repository already up-to-date
    • Filter matched zero repositories

Error Reporting

  • Detailed error messages with context
  • Aggregated error summary post-sync
  • Per-repository error state in TUI
  • Option to export error log

Performance Considerations

Optimization Strategies

  1. Caching

    • Repository metadata cache
    • GitHub API response caching
    • TTL-based invalidation

  2. Lazy Loading

    • Load repository details on demand
    • Paginate large repository lists
    • Stream results in TUI

  3. Shallow Clones

    • Configurable clone depth
    • Full clones for development repos
    • Shallow clones for reference

  4. Incremental Updates

    • Only fetch changed refs
    • Skip up-to-date repositories
    • Delta transfers

Security Considerations

Authentication

  • Leverage GitHub CLI’s authentication
  • Support SSH keys for Git operations
  • Never store credentials in config
  • Use system credential helpers

Safe Operations

  • Confirm before destructive operations
  • Detect uncommitted changes
  • Never force-push without explicit flag
  • Validate repository URLs
